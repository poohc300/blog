# 블로그 프로젝트 규칙

## 작업 진행 규칙

**모든 작업은 반드시 사용자 승인 후에만 시작한다.**

작업 요청을 받으면:
1. 구현 방향 및 변경 내용을 먼저 사용자에게 설명
2. 사용자가 명시적으로 승인한 후에만 작업 시작
3. 승인 없이 코드 작성, 파일 생성, DB 변경 등 어떠한 작업도 시작하지 않는다

## UI/디자인 변경 규칙

현재 확정된 UI 틀(레이아웃, 디자인, 반응형 구조)은 **절대 임의로 변경하지 않는다.**

변경이 필요한 경우:
1. 변경하려는 내용을 먼저 사용자에게 설명
2. 사용자 승인 후에만 진행

이 규칙은 UI뿐 아니라 레이아웃 구조, 색상 체계, 컴포넌트 스타일 전반에 적용된다.

## 블로그 글 작성 규칙

DB에 블로그 글을 작성하기 전에 반드시 기존 글 전체를 조회하여 문체와 스타일을 확인한다.

확인 절차:
1. `SELECT id, title FROM content ORDER BY created_at DESC;` 로 전체 글 목록 조회
2. 최근 글 2~3개의 body를 읽어 문체, 구성 방식, HTML 태그 사용 패턴 파악
3. 아래 문체 가이드와 대조하여 일관성을 유지하며 작성

### 문체 가이드 (기존 글 분석 결과)

**말투**
- 반말체, 건조하고 간결한 문장 (~다, ~했다, ~된다)
- 독자에게 설명하는 1인칭 시점 ("나는", "~가 필요했다", "~가 화근이었다")
- 군더더기 없이 핵심만. 불필요한 감탄이나 수식어 없음

**글 구성**
- 첫 단락: 문제 상황을 바로 서술. 배경 설명 최소화
- 에러 메시지나 문제 코드가 있으면 첫 단락 직후 `<pre><code>` 블록으로 바로 제시
- `<h2>` 섹션으로 구분 (h3은 쓰지 않음)
- 마지막 단락: 교훈 또는 인사이트 한 줄로 마무리

**섹션 제목 패턴**
- "원인" / "해결 방법" / "왜 이런 일이 생겼나" / "주의할 점"
- 또는 기술 용어 + 한국어: "Adjacency List 패턴", "Runner 설치"

**HTML 태그 규칙**
- 단락: `<p>`
- 섹션 제목: `<h2>`
- 코드 블록: `<pre><code>` (언어 class 없음)
- 인라인 코드: `<code>`
- 강조: `<strong>`
- 목록: `<ol>`, `<ul>` (필요한 경우만)
- `<br>`, 인라인 스타일 사용 금지

**기타**
- 기술 용어는 영어 그대로 (npm, JPA, nginx, SSR 등)
- 주어 생략 자유롭게
